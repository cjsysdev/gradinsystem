One road. One working website.

WS101 is built around a single client project — a real local business site — running on CodeIgniter 3. Every session adds one working piece to that same site. Nothing is a throwaway exercise.

The route starts on purpose from the browser side, not the framework side: build a form first, then ask the question that pulls the rest of the term into place.

ROUTE: html → forms → "where does submit go?" → ci3 → controller → model → database → auth → full crud → styled + shipped
01
WS101.01 — HTML FUNDAMENTALS

What can a page show with no server at all?

The browser draws something before any code runs on a server. What's actually happening?

builds: static homepage shell for the client site — structure only, no styling yet

  • tags & elements
  • semantic structure
  • headings & links
  • images
02
WS101.02 — HTML FORMS

How does a visitor hand information back?

A page can show things. How does it collect something from the person looking at it?

builds: a login form and an "add new item" form for the client site — plain HTML, not wired to anything yet

  • <form>
  • input types
  • labels
  • action & method
03
WS101.03 — THE PIVOT: WHERE DID SUBMIT GO?

You clicked submit. Where did that request actually land?

Watch the URL bar change on submit. Trace the action attribute. Hit a 404. Ask: something has to be sitting there to catch this — what is it?

builds: a traced, diagrammed path from the form's action to an empty destination — sets up the need for a framework

  • GET vs POST
  • action / method
  • index.php as the single entry point
04
WS101.04 — CI3 FILE & DIRECTORY STRUCTURE

So what's supposed to catch that request?

Something needs to live on the server and answer. Where does CodeIgniter expect that "something" to be?

builds: CI3 installed and running; the client project placed inside application/

  • MVC folders
  • application/ vs system/
  • request flow overview
05
WS101.05 — ROUTING

How does CI3 decide which code handles this specific submit?

There can be many forms and many pages. How does one URL get matched to one piece of code?

builds: the login form's action now points to a real, working route

  • routes.php
  • URI segments
  • default controller
06
WS101.06 — CONTROLLERS

The form arrived. What actually reads it?

A route points somewhere. What's sitting at that destination, and how does it get at what the user typed?

builds: a controller method that receives the form's POST data and echoes it back to prove the pipe works

  • controller structure
  • $this->input->post()
  • loading a view
07
WS101.07 — MODELS & THE DATABASE

The data can't just vanish after the page reloads. Where does it live?

The controller has the data for one request. What holds onto it after that?

builds: a model that saves the form's data into a real MySQL table for the client's core entity

  • Query Builder
  • insert / get / update / delete
  • table design basics
08
WS101.08 — VIEWS & DISPLAYING DATA BACK

The client wants to see the list they just built.

Data is saved now — but saved data no one sees is invisible. How does it get back on screen?

builds: a listing page that pulls real rows from the database and displays them

  • passing data to views
  • loops in views
  • PHP short echo tags
09
WS101.09 — SESSIONS & LOGIN

The login form works. Does the site remember who's logged in?

Submitting a login form is not the same as staying logged in. What keeps that state alive across pages?

builds: working login/logout with a session-protected admin page

  • CI3 Session library
  • password_hash / verify
  • role-based restriction
10
WS101.10 — THE FULL CRUD LOOP

Add, edit, delete — can the client run their catalog from the browser alone?

Saving one form is not a working site. Can every action a real owner needs be done without touching the database directly?

builds: add, edit, and delete all wired end-to-end on the client's core entity

  • update / delete methods
  • confirm dialogs
  • re-render after write
11
WS101.11 — STYLING & POLISH

It works. Does it look like something a client would trust?

Function is done. What separates a working page from a site a real business would put its name on?

builds: a responsive, Bootstrap-based layout applied across every page of the client site

  • grid system
  • cards / tables / modals
  • responsive breakpoints
12
WS101.12 — FINAL INTEGRATION

Can you explain, end to end, what happens when someone uses this site?

Trace one action — a client adding a new item — all the way from the form to the database and back to the screen. Every stage on this road should be a step in that trace.

builds: the complete, tested client website — live demo

  • full-stack request tracing
  • bug triage
  • client-facing presentation

// why this order

Starting with a form instead of a folder structure gives students a real question before they ever see CI3: "where did that go?" Every later stage — routing, controllers, models — exists to answer a question the form itself already raised, so the framework never feels arbitrary. By Stage 12, nothing on the site was built as a disconnected exercise; it's all one client project, one trace, start to finish.
